🤔 왜 플로우차트가 필요한가?

🗺️
전체 흐름을 한눈에
앱에서 사용자가 어떤 경로로 움직이는지 도식화하여, 팀 전체가 같은 그림을 공유할 수 있다.
🐛
논리 오류를 미리 발견
코딩 전에 흐름을 그려보면 "이 경우엔 뭘 보여줘야 하지?" 같은 빈 틈을 빠르게 발견할 수 있다.
🤝
협업 & 소통 도구
개발자, 디자이너, 기획자가 서로 다른 언어를 쓸 때도 플로우차트는 공통 언어가 된다.
⏱️
개발 시간 단축
설계가 확실할수록 코딩 중에 "이건 어떻게 하지?" 하고 멈추는 일이 훨씬 줄어든다.
🔄
수정이 쉽다
코드를 바꾸는 것보다 도형 하나를 옮기는 게 훨씬 빠르다. 설계 단계에서 마음껏 실험하자.
📋
문서화 & 유지보수
나중에 기능을 추가하거나 버그를 고칠 때, 플로우차트가 있으면 어디를 건드려야 할지 바로 보인다.

플로우차트 만드는 순서
01
앱의 핵심 기능을 목록으로 정리한다
예: 회원가입, 로그인, 상품 목록 보기, 장바구니 담기, 결제하기
02
사용자 입장에서 "처음에 뭘 하지?" 를 시작점으로
앱 실행 → 스플래시 화면 → 로그인 여부 확인 → 분기...
03
모든 Yes / No 분기를 빠짐없이 그린다
로그인 성공 → 메인, 로그인 실패 → 에러 메시지 → 재시도 or 비밀번호 찾기
💡 TIP: "이 상황에서 다른 케이스는 없나?" 를 항상 자문하세요.
04
팀원과 공유하고 빠진 흐름 보완
다른 사람에게 보여줬을 때 "이 경우엔?" 질문이 나오면 그게 빠진 분기다.
05
도구에 옮겨 정리 후 개발팀과 공유
draw.io, Figma, Miro 등에 정리해서 공식 문서로 남긴다.

PART 2 — Wireframe
와이어프레임

🤔 왜 와이어프레임을 만드는가?

📐
레이아웃을 확정한다
버튼은 어디에? 텍스트는? 이미지는? 색깔 없이 배치만 먼저 잡으면 나중에 디자인이 훨씬 쉽다.
🚫
디자인 낭비를 막는다
예쁘게 꾸몄는데 "여기에 버튼이 필요해요" 라는 말을 들으면? 와이어프레임 먼저 확인받으면 수정이 쉽다.
👁️
클라이언트에게 미리 보여준다
비전문가에게 플로우차트는 어렵지만 와이어프레임은 "이 화면이 이렇게 생겼어요" 로 바로 소통된다.
🧩
UX 흐름 검증
실제로 사용자가 화면을 넘기며 원하는 정보에 도달하는지, 동선이 자연스러운지 미리 테스트할 수 있다.
⚡
개발 속도 향상
개발자에게 "이 화면은 이렇게 구성하세요" 라고 보여주면, 스펙 회의가 줄고 오해가 없어진다.
🎯
핵심에 집중
색, 폰트, 이미지 없이 구조만 보이니까 "이 기능이 정말 필요한가?" 를 더 객관적으로 판단할 수 있다.
와이어프레임의 3가지 수준
🖊️
Lo-Fi 로우파이
종이에 손으로 스케치.
빠르고 유연. 아이디어 단계.

💻
Mid-Fi 미드파이
Figma, Balsamiq 등으로 작성.
회색 박스 & 텍스트 위주.
일반적 사용
🎨
Hi-Fi 하이파이
실제 디자인에 가까운 수준.
색상·폰트·이미지 포함.
프로토타입 직전

🎨 Figma로 와이어프레임 만드는 방법

01
Figma 접속 & 새 파일 만들기
figma.com 에 접속 → 로그인 → New design file 클릭.
무료 플랜으로도 충분히 와이어프레임을 만들 수 있다.
💡 Figma는 브라우저에서 바로 실행되므로 설치 불필요!
02
Frame(프레임) 으로 화면 크기 잡기
단축키 F → 오른쪽 패널에서 Phone (375×812) 또는 Desktop (1440×900) 선택.
각 화면(홈, 로그인, 상세 등)마다 별도의 Frame을 만든다.
03
Rectangle로 UI 요소 배치하기
단축키 R → 사각형 그리기. 버튼, 이미지, 카드, 네비게이션 바 등을 회색 박스로 대체해서 배치. 색깔은 지금 신경 쓰지 않는다.
💡 와이어프레임의 핵심은 "구조와 위치" 만 잡는 것!
04
Text 도구로 레이블 달기
단축키 T → 박스 안에 "버튼명", "상품명", "이미지 영역" 등 텍스트 표시.
폰트 크기로 헤더 / 본문 / 캡션의 위계를 표현한다.
05
Auto Layout으로 간격 자동 정렬
요소들을 선택 후 Shift + A → Auto Layout 적용.
버튼 패딩, 리스트 간격이 자동으로 맞춰져서 깔끔하게 정렬된다.
💡 Auto Layout = Figma의 핵심 기능! 꼭 익혀두자.
06
Prototype으로 화면 연결하기
오른쪽 패널 Prototype 탭 → 버튼을 다음 Frame에 연결 → ▶ Present 버튼으로 클릭 흐름 미리보기.
💡 실제로 터치해보는 것처럼 UX를 확인할 수 있다!
07
공유 링크로 피드백 받기
오른쪽 상단 Share → Copy link → 팀원이나 클라이언트에게 링크 전달.
상대방이 Figma 계정 없이도 브라우저로 확인하고 댓글을 달 수 있다.

플로우차트 vs 와이어프레임 비교

구분 🔷 플로우차트 🟡 와이어프레임
목적 앱의 로직·흐름 표현 화면의 구조·레이아웃 표현
보여주는 것 조건 분기, 사용자 이동 경로 버튼, 이미지, 텍스트 위치
사용 도구 draw.io, Miro, 종이 Figma, Balsamiq, 종이
언제 만드는가 기획 초기 단계 플로우차트 확정 후
주 사용자 개발자, 기획자 디자이너, 클라이언트, 기획자
수정 용이성 도형 위치만 바꾸면 됨 박스 옮기기, 색 없으니 빠름

✅ 오늘 수업 핵심 정리

  • 앱 개발의 순서: 아이디어 → 플로우차트 → 와이어프레임 → 디자인 → 개발
  • 플로우차트는 앱의 논리 흐름과 조건 분기를 도형으로 표현한다
  • 기본 도형: 타원(시작/끝), 사각형(처리), 다이아몬드(조건), 평행사변형(입출력)
  • 와이어프레임은 색깔 없이 화면 구조와 레이아웃만 잡는 설계도이다
  • Figma에서 F(프레임) → R(사각형) → T(텍스트) → Shift+A(Auto Layout) 순으로 작업
  • Prototype 기능으로 화면을 연결해 실제처럼 클릭 흐름을 확인할 수 있다
  • 둘 다 수정 비용이 거의 없다 — 코딩 전에 마음껏 실험하자!

✏️ 오늘의 실습 과제

1 draw.io 에서 "회원가입 → 로그인 → 메인 화면" 플로우차트 만들기 (조건 분기 포함)
2 Figma에서 내가 만들 앱의 주요 3개 화면 와이어프레임 작성하기
3 Figma Prototype 기능으로 3개 화면을 연결하고 링크 공유해보기
