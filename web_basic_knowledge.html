🍕 먼저 비유로 이해해봐요

인터넷은 피자 주문 시스템과 똑같이 작동해요.

🧑 손님(브라우저) → 📞 전화(HTTP 요청) → 🏪 피자집(서버)

손님이 "피자 주세요" 라고 요청하면
피자집이 "네, 여기 있어요!" 라고 응답하는 것처럼

브라우저가 "데이터 주세요" (요청, Request) 하면
서버가 "네, 여기 데이터요!" (응답, Response) 해요.

🤔 비동기가 왜 필요할까요?

서버에서 데이터를 가져오는 건 시간이 걸려요. 인터넷 속도에 따라 0.1초~수 초가 걸릴 수 있어요.
그 동안 브라우저가 완전히 멈춰있으면 어떻게 될까요?

☕ 카페 주문 비유

동기(나쁜 방식): "아메리카노 주세요" → 직원이 커피 만들 때까지 카운터에서 꼼짝 못 하고 기다림 → 다음 손님은 주문도 못 함

비동기(좋은 방식): "아메리카노 주세요" → 번호표 받음 → 자리에 앉아서 폰 봄 → "34번 손님 커피 나왔습니다!" → 가서 받아옴 ✅

📝 코드로 보는 비동기 — 3가지 방식

JavaScript에서 비동기 처리는 3단계로 발전했어요:

① 콜백 방식 (옛날 방식, 복잡해요)

// 콜백 지옥 — 중첩이 깊어질수록 읽기 어려워요 getData(function(result) { processData(result, function(result2) { saveData(result2, function(result3) { console.log('완료!'); // 3단계 중첩 → 읽기 힘들어요 😵 }); }); });

② Promise 방식 (개선됨)

// .then() 으로 순서대로 연결해요 getData() .then(result => processData(result)) .then(result2 => saveData(result2)) .then(() => console.log('완료!')) .catch(error => console.log('오류:', error)); // 에러 처리

③ async/await 방식 (현재 표준 ⭐) — Firebase에서 이걸 써요!

// 마치 순서대로 실행되는 것처럼 읽기 쉬워요 const loadData = async () => { const result = await getData(); // 기다렸다가 받아요 const result2 = await processData(result); // 순서대로! await saveData(result2); console.log('완료!'); // 훨씬 읽기 쉽죠? 😊 };
💡 async = "이 함수는 비동기야!" 선언 | await = "여기서 기다려!" 라는 뜻이에요.

🚨 try-catch — 오류 처리도 꼭 배워야 해요

서버 요청이 실패할 수도 있어요. 오류가 났을 때 앱이 완전히 멈추지 않도록 처리해요:

const getUserData = async () => { try { // 여기서 성공하면 실행돼요 const data = await getData(); console.log('성공!', data); } catch (error) { // 오류가 나면 여기로 와요 (앱이 멈추지 않아요) console.log('오류 발생:', error.message); } };
🎣 낚시 비유: try = 낚싯대를 던지는 것 | catch = 물고기 대신 신발이 걸렸을 때 처리하는 것
신발이 걸려도(오류가 나도) 낚시를 계속할 수 있어요!

🔥 Firebase에서 async/await가 이렇게 쓰여요!

const addUser = async () => { // async 꼭 붙여요 try { const docRef = await addDoc( // await: 저장될 때까지 기다려요 collection(db, 'users'), { name: '홍길동', age: 25 } ); console.log('저장 완료!', docRef.id); } catch (error) { console.log('저장 실패:', error); // 실패해도 앱은 살아있어요 } };

📋 JSON이란?

JSON(제이슨)은 JavaScript Object Notation의 줄임말이에요. 서버와 브라우저가 데이터를 주고받을 때 쓰는 공통 언어예요.

📦 택배 상자 비유:

물건을 보낼 때 아무렇게나 넣으면 안 되고, 정해진 박스에 정해진 방식으로 포장해야 배달이 돼요. JSON은 데이터를 담는 표준 포장 방식이에요!

🔍 JSON 구조 살펴보기

{
"name": "홍길동", ← 문자열은 쌍따옴표 안에
"age": 25, ← 숫자는 따옴표 없이
"isStudent": true, ← true / false
"nickname": null, ← 없음/비어있음
"hobbies": ["독서", "코딩"], ← 배열(리스트)
"address": { ← 객체 안에 객체
"city": "대전",
"zipcode": "34000"
}
}

📌 JSON 핵심 규칙 4가지

1

키(Key)는 반드시 쌍따옴표 " "

✅ "name": "홍길동"    ❌ name: "홍길동" (따옴표 없으면 오류!)

2

문자열 값도 쌍따옴표 " "

✅ "city": "서울"    ❌ "city": 서울

3

마지막 항목에는 쉼표 없이

✅ "age": 25 } (마지막엔 쉼표 없음)    ❌ "age": 25, }

4

주석을 쓸 수 없어요

❌ // 이런 주석 못 써요 — JSON 파서가 오류를 내요

🔄 JavaScript에서 JSON 변환하기

데이터를 주고받을 때 변환이 필요해요:

// JavaScript 객체 → JSON 문자열 (서버로 보낼 때) const user = { name: '홍길동', age: 25 }; const jsonString = JSON.stringify(user); // 결과: '{"name":"홍길동","age":25}' ← 문자열이 됐어요 // JSON 문자열 → JavaScript 객체 (서버에서 받을 때) const received = '{"name":"홍길동","age":25}'; const obj = JSON.parse(received); console.log(obj.name); // '홍길동' ← 이제 . 으로 접근 가능해요
💡 Firebase를 쓰면 JSON 변환을 직접 할 일이 거의 없어요. 내부적으로 자동 처리해줘요! 하지만 데이터 구조를 이해하는 데 꼭 필요해요.

데이터를 저장하는 창고 — SQL vs NoSQL

🗄️ 데이터베이스란?

데이터베이스는 데이터를 체계적으로 저장하고 관리하는 창고예요.

📚 도서관 비유:

도서관에 책이 쌓여 있으면 찾기 어렵죠. 그래서 분류 번호를 붙이고, 서가에 정리하고, 목록을 만들어요.

데이터베이스도 마찬가지예요. 수백만 개의 데이터를 빠르게 찾고, 저장하고, 수정할 수 있도록 체계적으로 정리해놓은 시스템이에요.

📊 SQL(관계형) vs NoSQL — Firebase는 어떤 건가요?

🏛️ SQL (관계형 DB)

  • 엑셀 표처럼 행/열로 저장
  • 미리 구조(스키마)를 정해야 함
  • MySQL, PostgreSQL 등
  • 정해진 형식의 데이터에 강함
  • 복잡한 관계 처리에 좋음

🔥 NoSQL (Firestore가 이것!)

  • JSON처럼 자유로운 구조
  • 구조를 나중에 바꿔도 OK
  • Firebase Firestore, MongoDB 등
  • 빠른 개발, 유연한 데이터에 강함
  • 실시간 동기화 처리에 좋음
💡 Firebase의 Firestore는 NoSQL이에요. 표 형태가 아닌 JSON 형태로 데이터를 저장해요!

🆚 SQL vs NoSQL 저장 방식 비교

같은 사용자 데이터를 어떻게 저장하는지 비교해봐요:

🏛️ SQL 방식 — 표(테이블)로 저장

id name age city
001홍길동25서울
002김철수30대전

🔥 NoSQL 방식 — JSON 문서로 저장 (Firestore)

// users 컬렉션 { "user_001": { ← 문서 ID name: "홍길동", age: 25, city: "서울", hobbies: ["독서", "코딩"] ← SQL엔 없는 배열도 가능! }, "user_002": { name: "김철수", age: 30, city: "대전" // hobbies가 없어도 괜찮아요 ← 유연해요! } }

⚡ 실시간 데이터베이스 vs 일반 데이터베이스

일반 DB vs 실시간 DB (Firestore) 차이

🐢 일반 DB 방식

1

사용자 A가 데이터 변경

2

사용자 B는 새로고침해야 확인 가능

⚡ Firestore 실시간 방식

1

사용자 A가 데이터 변경

2

사용자 B 화면에 즉시 자동 반영! 🎉

🔥 이게 바로 Firebase의 강력한 기능! 채팅앱, 실시간 협업 도구가 이 원리로 작동해요.
